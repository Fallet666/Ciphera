# Границы безопасности

## Что защищает E2EE

**End-to-End Encryption (E2EE)** защищает:
- Текст сообщений
- Файлы и вложения
- Фото и видео
- Голосовые сообщения
- Историю переписки

В зашифрованных комнатах **сервер не может прочитать** содержимое сообщений и файлов.

## Что НЕ защищает E2EE

E2EE **не скрывает**:
- Список аккаунтов на сервере
- IP-адреса подключений
- Время активности пользователей
- Размеры загружаемых файлов
- Состав комнат (кто с кем общается)
- Метаданные на уровне Matrix протокола

**Администратор сервера может:**
- Удалить базу данных
- Удалить медиа-файлы
- Удалить аккаунты
- Видеть метаданные

**Компрометация устройства:**
- Если злоумышленник получил доступ к устройству пользователя — он получит доступ ко всей переписке на этом устройстве

## Обязательные меры безопасности

### 1. Проверка устройств

Каждое устройство имеет свои ключи шифрования. При добавлении нового устройства:
- Проверьте его через **SAS verification** (сравнение эмодзи)
- Убедитесь, что это действительно ваше устройство или устройство члена семьи

### 2. Сильные пароли

- Минимум 12 символов
- Случайная комбинация слов, цифр, символов
- Уникальный пароль для каждого пользователя
- Использовать парольный менеджер

### 3. Закрытая регистрация

- Регистрация **только по одноразовым токенам**
- После регистрации всех членов семьи — **отключить регистрацию полностью**
- Не публиковать ссылку на чат публично

### 4. Recovery key

- Сохранить в **нескольких безопасных местах**
- Бумага, парольный менеджер, флешка
- **Никогда не пересылать** через другие каналы связи
- Без recovery key невозможно восстановить доступ к зашифрованным сообщениям

### 5. Регулярные обновления

- Обновлять Synapse, Element, coturn еженедельно
- Обновлять ОС сервера
- Проверять логи на подозрительную активность

### 6. Зашифрованные бэкапы

- Бэкапы PostgreSQL + media_store + configs
- Шифровать перед отправкой на удалённый сервер
- Хранить минимум 14 дней
- Проверять возможность восстановления

### 7. Минимизация логов

- Не включать debug-логи
- Настроить logrotate
- Хранить логи минимум времени
- Не логировать request body

## Угрозы и защита

### Brute-force атаки

**Защита:**
- fail2ban для SSH и nginx
- Rate limiting на login/register
- Сильные пароли
- Закрытая регистрация

### DDoS атаки

**Важно:** на одном VPS невозможно защититься от volumetric DDoS.

**Защита от application-layer:**
- nginx rate limiting
- fail2ban
- firewall
- Закрытая регистрация
- Отключённая federation

### Компрометация сервера

**Защита:**
- Регулярные обновления
- Минимальные права доступа
- Зашифрованные бэкапы на удалённом сервере
- Мониторинг логов

### Компрометация устройства

**Защита:**
- Проверка устройств через verification
- Немедленное удаление скомпрометированных устройств
- Recovery key для восстановления

## Что НЕ делать

- **Не писать свой E2EE-протокол** — использовать проверенный Matrix/Olm/Megolm
- **Не хранить plaintext сообщений** на сервере
- **Не включать открытую регистрацию**
- **Не давать один аккаунт нескольким людям**
- **Не использовать простые пароли**
- **Не оставлять debug-логи** включёнными
- **Не публиковать admin-токен**
- **Не запускать Synapse без PostgreSQL**
- **Не включать federation** для семейного сервера
- **Не полагаться на один сервер** как на единственный бэкап

## Мониторинг

Регулярно проверяйте:
- `ciphera health` — статус сервисов
- Логи fail2ban — попытки взлома
- Логи nginx — подозрительные запросы
- Дисковое пространство — заполнение media_store

## Инциденты

При подозрении на компрометацию:

1. **Немедленно** смените пароли всех пользователей
2. Удалите подозрительные устройства через админ-панель
3. Проверьте логи на аномалии
4. Восстановите из бэкапа при необходимости
5. Рассмотрите возможность пересоздания сервера
